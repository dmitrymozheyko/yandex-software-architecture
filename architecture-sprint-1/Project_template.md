### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователи могут включать/выключать отопление
- Система поддерживает добавление/редактирование/удаление устройств управления отоплением

**Мониторинг температуры:**

- Пользователи могут получать значение температуры с датчиков
- Система поддерживает добавление/редактирование/удаление датчиков температуры

### 2. Анализ архитектуры монолитного приложения

- Язык программирования: Go
- База данных: PostgreSQL
- Архитектура: Монолитная, все компоненты системы (обработка запросов, бизнес-логика, работа с данными) находятся в рамках одного приложения.
- Взаимодействие: Синхронное, запросы обрабатываются последовательно.
- Масштабируемость: Ограничена, так как монолит сложно масштабировать по частям.
- Развертывание: Требует остановки всего приложения.

### 3. Определение доменов и границы контекстов

- Домен: Управление пользователями
  - Поддомен: Администрирование клиентских договоров
    - Контекст: регистрация нового клиента
    - Контекст: блокирование клиента
- Домен: Личный кабинет пользователя
  - Поддомен: Администрирование набора устройств
    - Контекст: Администрирование активных устройств
    - Контекст: Администрирование сенсоров телеметрии
    - Контекст: Регистрация нового устройства в системе
  - Поддомен: Управление активными устройствами (реле)
    - Контекст: Получение текущего состояния
    - Контекст: Изменение состояния
  - Поддомен: Телеметрия (получение данных с сенсоров)

### **4. Проблемы монолитного решения**

- монолитное решение не приспособлено для покомпонентного масштабирования, все функции работают в рамках одного процесса 
- монолитное решение имеет единую точку отказа, т.е. при сбое мы теряем работоспособность всей системы
- невозможно покомпонентное развёртывание/обновление малой части функций
- синхронное взаимодействие с удалёнными частями системы может вызывать проблемы с производительностью при сетевых задержках
- все компоненты приложения должны быть реализованы в рамках одного стека, нет возможности поделить части системы между гетерогенными командами (разработчиками)
